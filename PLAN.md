## 1a. Архитектура UX — wizard «Начать тхун» (решено 2026-05-13)

Ключевое решение: Подготовка / Медитация / Завершение **не разнесены по разным вкладкам**, а собраны в **единый wizard**, запускаемый кнопкой «Начать тхун» на экране Журнала. Это даёт интуитивный, линейный поток для реальной практики:

```
ГЛАВНАЯ (= Журнал)  →  [ Начать тхун ]
                           │
                           ▼
   Шаг 1 · Приготовления   (checklist: место, тело, инструменты)
                           │
                           ▼
   Шаг 2 · Подготовка      (9 дыханий → шаматха [6 мин, регулируемая] →
                            молитвы Лама Чхенно / прибежище / 4 безмерных /
                            Бодхичитта / семистрочная)
                           │
                           ▼
   Шаг 3 · Обет            (форма: длительность X, место, признак →
                            текст обета формируется и читается)
                           │
                           ▼
   Шаг 4 · Медитация       (таймер, литургия признака на экране,
                            ±круг, отдохновение, отображение прогресса)
                           │
   ⏰ → бэлл звенит         (Web Audio, мягкий gong, при истечении таймера)
                           │
                           ▼
   Шаг 5 · Завершение      (Самопроверка: ошибки/успехи →
                            Посвящение: устремление/обещание/молитва)
                           │
                           ▼
   Шаг 6 · Сохранить       → запись в Журнал, прогресс обновляется
```

**Параллельный путь — Quick Log**: существующая простая форма «Добавить тхун» остаётся для случаев, когда сессия не записана в реальном времени.

**Структура вкладок**: 4 (без изменений по количеству):
- `Журнал` (главная, с кнопкой «Начать тхун» и недельной статистикой)
- `Прогресс`
- `Практика нёндро` (бывший `Текст` — литургия + recitations для чтения вне сессии)
- `Настройки`

Wizard живёт как **модальный overlay** (full-screen), а не как пятая вкладка.

## 3. Миграция данных (обратная совместимость)

В `loadState()` добавляется блок:

```js
const GROUPKEY_MIGRATION = {
  // s2
  "impermanence": "impermanenceExamples",      // дефолт — на «примеры»
  // s3
  "samsaraGeneral": "samsaraPoisons",
  "sixRealms": "samsaraPoisons",
  // s4
  "nonVirtuous": "karma",
  "virtuous": "karma",
  "karmaGeneral": "karma",
  // s5
  "liberationCause": "nirvanaFreedoms",
  "liberationFruit": "nirvanaFreedoms",
};
state.tuns.forEach(t => {
  if (GROUPKEY_MIGRATION[t.groupKey]) {
    t.groupKey = GROUPKEY_MIGRATION[t.groupKey];
    t.idx = 0; // сбрасываем idx — items в новой группе не совпадают
  }
});
```

Это даёт обратную совместимость, не теряя записей. Простой даунгрейд (export → ручная правка) описать в README.

## 9b. Будущая работа · cloud sync на Vercel (зафиксировано 2026-05-13)

**Проблема**: localStorage привязан к (origin, device). Деплой на Vercel сам по себе **не решает** кросс-девайс синхронизацию — Safari на телефоне и Chrome на ноутбуке по одному URL имеют разные хранилища. JSON export/import — это UX-тупик, не подходит для практикующих и тем более для не-технических пользователей.

**Решение (target by 2026-05-27)**:

| Слой | Выбор | Бесплатно? |
|---|---|---|
| Хостинг | Vercel | да |
| Auth | Clerk (Google OAuth + email magic-link) | да до 10k MAU |
| База | Vercel KV или Neon Postgres | да в рамках лимитов |
| Sync | `/api/sync` GET + POST в существующем HTML | — |

**Модель данных**: каждый пользователь — отдельная Clerk-учётка, его `state.tuns` хранится в KV по `userId`. У Ильи свой деплой со своими пользователями, у друга — свой со своими. Никакого пересечения данных.

**UX**:
1. Открываешь `ngondro.ohmyseas.com` → видишь кнопку «Войти» (один клик Google).
2. После входа данные приходят с сервера, дальше любое сохранение пишет и локально, и на сервер (debounced).
3. На любом другом устройстве — войти той же Google-учёткой → видишь свои тхуны.
4. Не-технические пользователи: один клик Google — никаких токенов, никаких PAT, никакого JSON.

**Когда строить**: после 2–3 реальных тхунов на текущей версии (полевое тестирование UX wizard'а). Если в wizard всплывут проблемы — фиксим до миграции, а не после.

**Что войдёт в этот этап**:
- [ ] Создать новый Vercel проект из репозитория `ohmyseas/ngondro`
- [ ] Подключить Clerk (DEV + PROD ключи)
- [ ] Создать Vercel KV или Neon Postgres
- [ ] Добавить `api/sync.js` (GET/POST handlers)
- [ ] Добавить sync-layer в HTML (~150 lines: auth state, debounced save, merge-on-load)
- [ ] Кнопка «Войти» в верхней панели; если не вошёл — режим «только локально» (как сейчас)
- [ ] Конфликт-резолюшн: append-only по `tun.id`; при коллизии — last-write-wins для отдельного тхуна
- [ ] README инструкция для самохостинга (если друг захочет свой деплой)

**Альтернатива A — GitHub Gist sync**: отвергнута 2026-05-13. Работает для технических, но требует PAT — слишком крутая кривая для не-технических пользователей, которым в будущем может понадобиться доступ.

**Альтернатива B — Nextcloud WebDAV (rita-server)**: отложена. Имеет смысл как личный extra-layer privacy, но не решает multi-user. Можно добавить опционально позже.

## 10. Risks / зачем это может не взлететь

- **Объём текста**: 34 × 14 + 26 × ~14 + остальное ≈ 1200+ строк литургии. HTML вырастет с 74 KB до ~150–200 KB. Всё ещё ок для single-file, но граница приличий.
  - *Митигация*: текст в `<script>` как JS-константы, не как HTML.
- **localStorage у друга**: если у него реальные сохранённые тхуны под старыми `groupKey`, миграция должна сработать корректно. Проверим на его JSON-экспорте перед merge обратно.
- **Согласование с лайнэйджем**: я не учитель. Если в текстах PDF есть опечатки или редакторские варианты — копирую дословно, любые правки только по указанию друга.
